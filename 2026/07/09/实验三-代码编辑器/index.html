    实验三：代码编辑器

    1. 实验目的

    本次实验旨在完成以下核心目标：

    1. 安装并配置 Visual Studio Code 代码编辑器：掌握从官网下载、安装、配置环境变量的完整流程，理解代码编辑器与集成开发环境（IDE）的区别与联系。

    2. 掌握通过扩展面板安装 Markdown 相关插件的方法：熟悉 VS Code 的插件生态系统，能够根据需求自主搜索、安装、配置开发插件，理解“编辑器 + 插件”的轻量化开发理念。

    3. 学习 Markdown 基础语法并使用实时预览功能编写结构化文档：掌握 Markdown 的核心语法（标题、列表、代码块、粗体、链接等），能够使用 VS Code 的预览功能进行“所见即所得”的文档写作。

    4. 完成 Git 全局信息配置与本地仓库初始化：掌握 Git 版本控制系统的基础配置方法，理解 user.name 和 user.email 在提交记录中的重要作用，能够独立初始化本地 Git 仓库并完成首次提交。

    5. 实现项目代码库的基础版本控制：通过实际操作理解 Git 的“工作区 → 暂存区 → 本地仓库”三区工作流，为后续团队协作和持续集成奠定基础。

    3. 实验过程

    步骤一：下载并安装 Visual Studio Code

    3.1.1 下载安装包

    打开浏览器，访问 VS Code 官方网站（https://code.visualstudio.com/）。VS Code 由微软开发并开源，是目前全球最受欢迎的轻量级代码编辑器，以其丰富的插件生态、优秀的性能和跨平台支持而著称。

    在官网首页，点击“Download for Windows”按钮，下载最新版本的 Windows 安装包。安装包大小约为 80-100MB，下载速度取决于网络状况。

    3.1.2 运行安装向导

    双击下载的 .exe 安装包文件，启动安装向导。Windows 可能会弹出“用户账户控制”提示框，点击“是”允许安装程序运行。

    在安装向导的欢迎界面，点击“下一步”继续。在“许可协议”页面，阅读并接受许可协议，勾选“我接受许可协议中的条款”，点击“下一步”。

    3.1.3 选择安装路径

    在“选择目标位置”页面，可以选择默认安装路径（通常为 C:\Program Files\Microsoft VS Code\）或自定义路径。建议使用默认路径，方便后续查找和维护。点击“下一步”。

    3.1.4 选择附加任务（关键步骤）

    在“选择附加任务”页面，这是整个安装过程中最关键的一步。务必勾选以下选项：

    选项 重要性 说明
    创建桌面快捷方式 ⭐⭐⭐ 方便快速启动 VS Code
    将“通过 Code 打开”添加到资源管理器文件上下文菜单 ⭐⭐⭐⭐ 在任意文件上右键可直接用 VS Code 打开
    将“通过 Code 打开”添加到资源管理器目录上下文菜单 ⭐⭐⭐⭐ 在任意文件夹上右键可直接用 VS Code 打开
    将 Code 注册为受支持的文件类型的编辑器 ⭐⭐⭐ 双击代码文件自动用 VS Code 打开
    添加到 PATH（必须勾选） ⭐⭐⭐⭐⭐ 勾选后可在任意终端中直接使用 code 命令，这是后续开发工作流的基础

    其余选项保持默认，点击“下一步”。

    3.1.5 完成安装

    点击“安装”按钮，等待安装进度条完成（约需 2-3 分钟）。安装完成后，点击“完成”退出安装向导。如果勾选了“启动 Visual Studio Code”，安装完成后会自动启动编辑器。

    验证安装：安装完成后，打开命令提示符或 PowerShell，输入 code --version，如果能够正常输出版本号，说明安装成功且 PATH 环境变量配置正确。

    3.2.1 启动 VS Code

    双击桌面上的 VS Code 快捷方式图标，启动编辑器。首次启动时，会显示欢迎页面（Welcome Page），其中包含最近打开的项目、官方文档链接、教程视频等。可以将其关闭，或勾选“Don’t show again”避免下次显示。

    启动后，熟悉 VS Code 的主界面布局：

    区域 名称 功能说明
    左侧最窄竖条 活动栏 包含资源管理器、搜索、源代码管理（Git）、运行和调试、扩展等核心功能图标

    任务列表 支持 - [ ] 和 - [x] 语法
    自动编号 支持有序列表的自动编号

    功能 说明

    数学公式 支持 LaTeX 数学公式渲染（$…$ 和 $$…$$）

    这是一个 Markdown 语法检查工具：

    功能 说明
    语法检查 检查 Markdown 文件中的语法错误和格式问题
    风格规范 遵循 Markdown 社区的最佳实践和规范

    步骤三：配置编辑器核心设置

    3.3.1 打开设置面板

    按 Ctrl + ,（逗号）打开设置面板。VS Code 的设置分为两个级别：

    级别 说明 适用场景
    用户设置 对所有项目生效 个人偏好配置，如主题、字体、自动保存等
    工作区设置 仅对当前项目生效 项目特定配置，如项目专属的格式化规则、Python 解释器路径等

    本次实验配置用户级别设置，以便在所有项目中复用。

    3.3.2 配置自动保存

    在设置搜索框中输入 files.autoSave，在下拉选项中选择 **afterDelay**。

    功能说明：开启自动保存后，VS Code 会在用户停止编辑的短暂延迟（默认 1000 毫秒）后自动保存文件。这一功能具有以下优势：

    • 避免内容丢失：防止因忘记手动保存（Ctrl+S）而导致的代码或文档内容丢失
    • 提升工作效率：减少手动保存的操作次数，专注于内容创作
    • 配合实时预览：自动保存配合 Markdown Preview Enhanced 的自动刷新，实现真正的“所见即所得”

    其他可选值：

    • off：关闭自动保存（需要手动 Ctrl+S 保存）
    • onFocusChange：编辑器失去焦点时自动保存
    • onWindowChange：窗口失去焦点时自动保存

    3.3.3 配置自动换行

    在设置搜索框中输入 editor.wordWrap，在下拉选项中选择 **on**。

    功能说明：开启自动换行后，长文本会在编辑器视口边缘自动换行显示，无需水平滚动条即可阅读完整内容。

    适用场景：

    • Markdown 文档写作（段落通常较长）
    • README 文件编辑
    • 代码注释阅读
    • 日志文件查看

    其他可选值：

    • off：关闭自动换行（需要水平滚动查看长行）
    • on：根据视口宽度自动换行
    • wordWrapColumn：在指定列数处换行
    • bounded：在视口宽度和指定列数中较小值处换行

    3.3.4 其他推荐配置

    除了上述两个核心配置外，以下配置也有助于提升开发体验：

    配置项 推荐值 说明
    editor.fontSize 16 适当增大字体大小，保护视力，减少眼睛疲劳
    workbench.colorTheme Dark+ (default dark) 选择深色主题，长时间工作更舒适
    editor.minimap.enabled true 开启缩略图，方便快速定位文档位置
    editor.renderWhitespace all 显示空格和制表符，便于排查缩进问题
    files.autoSaveDelay 1000 自动保存延迟时间（毫秒）

    步骤五：初始化 Git 本地仓库

    3.5.1 Git 仓库的概念

    Git 仓库（Repository）是 Git 版本控制系统的核心概念，它是一个被 Git 管理的项目文件夹，其中包含了项目的所有文件以及它们完整的修改历史记录。

    Git 仓库的主要组成部分：

    组成部分 说明
    .git 隐藏文件夹 存储所有版本信息的核心目录，包含 commit 对象、分支引用、配置等，任何情况下都不要手动修改或删除
    工作区（Working Directory） 当前正在编辑和查看的项目文件
    暂存区（Staging Area / Index） 准备提交的文件列表
    本地仓库（Local Repository） 存储所有历史提交记录

    3.5.2 打开项目文件夹

    在 VS Code 中，有两种方式打开项目文件夹：

    方式一：点击菜单栏“文件” → “打开文件夹”，在弹出的对话框中选择项目目录

    方式二：直接在文件资源管理器中右键点击项目文件夹 → “通过 Code 打开”

    步骤七：提交代码到本地仓库

    3.7.1 Git 三区工作流

    在执行提交之前，需要理解 Git 的核心工作流程——三区模型：

    第一步：工作区——我们在 VS Code 中编写和修改代码，这些修改位于工作区。

    第二步：暂存区——通过 git add 命令或点击 + 号，将确认无误的修改添加到暂存区，准备提交。

    第三步：提交——通过 git commit 命令或点击 ✔ 号，将暂存区的内容正式保存为版本快照。

    3.7.2 查看文件状态

    打开“源代码管理”面板（Ctrl + Shift + G），可以看到 README.md 出现在“更改”列表中，文件名旁显示字母 U（Untracked，未跟踪）。

    文件状态标识说明：

    标识 含义 说明
    U（Untracked） 未跟踪 新文件，Git 尚未管理
    M（Modified） 已修改 文件已被修改，但尚未暂存
    A（Added） 已暂存 文件已添加到暂存区，等待提交
    D（Deleted） 已删除 文件已被删除

    3.7.3 暂存文件

    在“源代码管理”面板中，鼠标悬停在 README.md 文件上，点击右侧出现的 +（加号） 按钮，将文件从工作区添加到暂存区。

    暂存后，文件会从“更改”区域移动到上方的“暂存的更改”区域。

    3.7.4 填写提交信息

    在“源代码管理”面板顶部的消息输入框中，输入本次提交的说明信息：

    提交信息规范说明：

    提交类型 前缀 使用场景
    新功能 feat: 添加新功能或新文件
    修复 Bug fix: 修复代码错误
    文档更新 docs: 更新文档或 README
    格式调整 style: 代码格式调整（不影响逻辑）
    性能优化 perf: 性能优化相关修改
    测试相关 test: 添加或修改测试代码

    首次提交通常使用 Initial Commit 作为提交信息，表示这是项目的第一个版本快照。

    3.7.5 执行提交

    点击消息输入框上方的 ✔（对勾） 按钮，执行提交操作。

    提交过程通常非常快速（毫秒级），提交成功后：

    1. “更改”和“暂存的更改”列表均清空
    2. 状态栏显示“无待提交更改”（没有数字角标）
    3. Git 已为当前项目状态创建了一个版本快照

    提交记录截图

    3.7.6 查看提交历史

    在扩展面板中安装 Git Graph 插件（已安装），可以图形化地查看提交历史。

    点击底部状态栏的分支名称（如 main），或在命令面板（Ctrl+Shift+P）中输入 Git Graph，即可打开图形化视图。

    从 Git Graph 中可以清晰看到：

    • 提交节点：显示提交信息、作者、时间
    • 分支结构：显示当前分支的演进路径
    • 提交哈希值：每个提交唯一的标识符

    知识点 分类 详细说明
    VS Code 插件机制 编辑器 通过扩展面板（Ctrl+Shift+X）搜索和安装插件，插件安装后通常立即生效，无需重启编辑器。VS Code 拥有全球最大的编辑器插件生态

    6. 出现问题及解决方案

    问题一：Git 配置时需确认是否正确

    问题现象：在配置 Git 全局信息时，不确定是否配置成功。

    排查过程：

    1. 执行 git config --global --list 查看所有全局配置
    2. 确认输出中包含 user.name 和 user.email

    原因分析：未配置 user.name 和 user.email 时，首次提交会报错“请告诉我你是谁”。这是 Git 的安全机制，要求每次提交必须有明确的作者身份标识，便于代码变更追溯和责任划分。

    问题二：VS Code 终端无法识别 git 命令

    问题现象：在 VS Code 终端中输入 git 命令，提示“无法识别”。

    可能原因及解决方案：

    1. Git 未安装：访问 git-scm.com 下载并安装 Git
    2. Git 未添加到 PATH：在安装 Git 时，选择“Git from the command line and also from 3rd-party software”选项
    3. 需要重启终端：安装 Git 或修改 PATH 后，需要重启 VS Code 终端才能生效
    4. VS Code 需要重启：有时需要完全重启 VS Code 才能识别新安装的软件

    技术层面

    通过本次实验，我全面掌握了 VS Code 这一现代化代码编辑器的安装、配置和使用方法，深刻理解了“编辑器 + 插件”的轻量化开发理念。与传统集成开发环境（IDE）的“开箱即用”但“大而全”不同，VS Code 只提供核心编辑功能，通过插件生态实现功能的按需扩展。这种设计既保持了编辑器的轻量高效（启动速度快、资源占用低），又具备了几乎无限的扩展能力，是一种非常优雅的软件架构设计。

    Markdown 的简洁语法让我印象深刻。仅需 #、-、*、``` 等少量符号，就能完成文档的结构化排版，让我能够**专注于内容本身而非排版细节**。在 VS Code 中配合实时预览功能（Ctrl+K V），真正实现了“所见即所得”的写作体验。这种“内容优先”的设计理念，与软件开发中“代码优先”的敏捷思想异曲同工，都强调直接面向核心价值进行创作。

    Git 的“修改 → 暂存 → 提交”工作流则让我初步建立了版本控制的规范意识。每一次提交都应该是一个原子性的、有明确目的的逻辑单元，提交信息应该清晰描述本次变更的内容和原因。这些看似“繁琐”的规范，实际上是团队协作效率的保障，也体现了软件工程中“先设计后编码”“先审阅后提交”的严谨作风。
